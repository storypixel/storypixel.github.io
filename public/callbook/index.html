What this is

Callbook turns a team's play calls into a living playbook. Plays are written in DBN (Dodgeball Notation) — a compact, human-writable notation in the spirit of chess algebraic notation. A whole play is a handful of speakable lines: who holds balls, where people run, who fakes, who throws at whom and what happened. A tiny engine (no build step, no dependencies, two script tags) compiles the notation into an animated overhead court you can embed in any web page.

You write intent; the engine owns aesthetics — formations, spacing, ball-flight curves, dodges and step-captions are computed, and every named word has exact, documented semantics. Nothing is magic: anything the animator can draw can be written, overridden, or escaped by hand.

Write plays

    
The court is a top-down chess diagram: THEM on top, US on the bottom. Players are
    U1…U8 and T1…T8 left to right (any team size). One
    token per action; run player digits together to fan an action across a group:

    

      
actiontoken

      
run to a named spotU3-line · U1458-huddle · U4-deep

      
grab loose ballsU8*2 (the two nearest)

      
pass / throwU8>U5 · U1@T3! (hit) % dodged ^ caught # blocked

      
pump fakeU3? · U3456?2 (two fakes in unison)

      
exact position escapeU5-68 · U4-(48,65) — always works

    

    
Formations are computed with real dodgeball sense: a parley is shoulder-to-shoulder, an
    attacking line spreads across the middle half, defensive depths spread wide, and retreats
    fall straight back — players never re-fan while regressing.

Put it on a page

    
Two script tags, then drop notation into any element — it auto-mounts:

    
<script src="https://iamnotsam.com/dodgeball-play-notation/vendor/play-animator.js"></script>
<script src="https://iamnotsam.com/dodgeball-play-notation/vendor/dbn.js"></script>

<div data-db-play-dbn data-loop>
[Play "Home"]
[Balls "U:45 T:2468"]
1. {They bring it up} T2468-line T4?
2. {Pre-counter} T4@U5% U5@T4!
</div>

    
Or self-host: fork storypixel/dodgeball-play-notation
    (MIT-spirited, plain JS, runs from any static host) and serve vendor/ yourself.
    DBN.parse(text) returns plain play JSON; DodgeballPlay.mount(el, play, opts)
    renders and returns a player controller.

  


  

    
What a team gets

    
    
      
  • A playbook page — plain document, one animated widget per call, a permalink
      button on every play so you can text a teammate straight to the right diagram.
      Live example: a real team's playbook ·
      a neutral demo.
    • 
      
  • A quiz — watch each play with no labels and name the call; 80% certifies,
      a perfect run earns a distinction, and the certificate copies to the clipboard as an
      image for the group chat. Live: playbook quiz.
    • 
      
  • One source of truth — the playbook, the quiz, and every embedded diagram
      read the same .dbn files. Fix a play once; every surface updates.

For AI agents

    
This system is built to be driven by agents. Machine-usable surfaces, all stable URLs:

    
    
      
  • Spec: /dodgeball-play-notation/NOTATION.md
      — ASCII, exact semantics for every token, EBNF grammar at the bottom.
    • 
      
  • Agent guide: /dodgeball-play-notation/DRIVING.md
      — the window.DBNEditor automation API (load, render, exportSVG,
      exportJSON, getErrors, player transport), plus test ids on every control.
    • 
      
  • Render any notation from a URL: /dodgeball-play-notation/?dbn=<url-encoded DBN>
      opens the editor view on your play; parse errors surface in #error-panel.
    • 
      
  • Play files: /dodgeball-play-notation/examples/<id>.dbn —
      plain text, fetchable; the compiled JSON carries per-beat summary captions.
    • 
      
  • Deep links: every play is an anchor on one page
      (/dodgeball-play-notation/#kill-left, /dodging-playbook/#mirror).
    • 
      
  • Headless: src/dbn-headless.js compiles DBN → JSON / static SVG in
      pure Node; the repo ships a CLI (validate, show, describe,
      link) and golden-fixture tests.
    • 
      
  • Start here: /callbook/llms.txt
    • 
    

    
House rule worth copying: no magic. Every engine capability ships as a notation
    token + a spec entry + a test, or it doesn't ship. Conventions (who fakes, how wide
    formations spread, how retreats move) are enforced by the test suite, so a play that
    violates team doctrine fails CI instead of confusing the next reader — human or agent.
